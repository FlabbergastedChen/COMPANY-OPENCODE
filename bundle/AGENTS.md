# Project Rules

适用范围：`bundle/` 目录及其子目录。

## Working Mode
1. 默认先理解目标、约束与验收标准，再开始实现。
2. 用户未指定流程时，按任务阶段选择一种主流程执行，不混用主路径：
   - 需求与规划优先：`speckit-*`
   - 变更生命周期管理优先：`openspec-*`
   - 工程执行与评审协作优先：`superpowers-*`
3. 如果上下文不足以安全执行，先说明缺失信息并请求补充。

## Execution Rules
1. 采用小步可验证的推进方式，保持阶段边界清晰（规划、实现、验证、收口）。
2. 并行仅用于独立问题域；共享写入面必须串行处理。
3. 不做无依据猜测；遇到阻塞时输出阻塞原因、影响和可选方案。
4. 所有改动应可追溯到明确任务或需求。

## Quality Gates
1. 无当前会话中的验证证据，不得给出“已完成”结论。
2. 评审反馈默认按风险分级，并包含影响范围与修复建议。
3. 高风险未闭环时，默认结论为不可合并或需继续修改。

## Instructions Loading
1. 本文件只保留项目级核心规则，避免重复通用规则。
2. 细分规则通过 `opencode.jsonc` 的 `instructions` 字段加载。
3. 外部规则按需组织，避免无关长文本进入默认上下文。

## Conflict Resolution
1. 优先遵循用户显式要求与任务目标。
2. 同层规则冲突时，选择更具体、可验证、可执行的规则。
3. 若仍冲突，明确列出选项与风险后再继续。
