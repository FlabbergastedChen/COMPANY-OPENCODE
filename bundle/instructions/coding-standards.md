# 编码规范（三框架提炼版）

## 实现风格
1. 以“满足已确认需求的最小正确改动”为优先。
2. 明确模块边界：编排逻辑、领域逻辑、副作用处理应分离。
3. 优先确定性行为与显式错误路径，避免隐式兜底。
4. 默认保持向后兼容；若必须破坏兼容，需明确声明并记录影响。

## 变更卫生
1. 每个改动都应能映射到已声明的需求或任务。
2. 保持 diff 可审查：不要把无关重构与功能修复混在一起。
3. 若存在假设，必须在输出中显式说明并标注风险级别。

## 可审查性
1. 关键路径要易于检查：输入校验、状态流转、失败处理。
2. 仅在意图不直观处添加简洁注释，避免噪音注释。
3. 显式说明运维影响：配置变化、迁移步骤、发布与回滚注意点。

## 安全约束
1. 未运行或未引用相关检查证据时，不得宣称成功。
2. 不得为追求速度绕过既有保护机制。
3. 修复不确定时，优先可回滚、影响面小的方案。
